# ADR 02 архитектура фронтенда

## Проблематика

Требования к архитектуре:

- независимость отдельных бизнес-юнитов (доставка, управление ресторанами, работа с курьерами);
- минимальное время рендера;
- отсутствие бизнес-логики на фронтенде;
- перераспределение ресурсов и отказоустойчивость.

## Обоснование

Монолитная структура мешает параллельной разработке независимых частей проекта — от кода до релизов, из‑за чего растут конфликты. Проект предполагает отдельные бизнес-юниты с возможностью интеграции.

Скорость работы сервиса — ключевая метрика для удержания пользователей, числа успешных заказов и привлечения бизнеса.

Архитектура бэкенда изначально такая: каждый продуктовый сценарий обрабатывает один сервис, он же отвечает за межсервисное взаимодействие. Для клиента используется единая GraphQL-схема ([ADR 03](./03.data-format-graphql.md)).

## Решение

- Отдельные SSR-сервисы под каждый бизнес-юнит.
- Без бизнес-логики на фронте: можно отказаться от серверной прослойки, делать жёсткую навигацию между сервисами без общего состояния, без event bus и прочей кросс-навигации, без оркестрации на клиенте.
- Стек: Next.js с App Router.

## Последствия

Отдельные сервисы друг о друге не знают (интеграция через iframe или module federation — отдельно, там много ограничений). Весь фронтенд нужно сделать ленивым ради UX.

### Плюсы от внедрения

- SSR: короче time-to-market, лучше наблюдаемость, независимые релизы частей приложения, меньше нагрузки на клиент.
- Не нужна оркестрация на клиенте.
- Масштабируемость, QoS, наблюдаемость.
- Next даёт кэширование и навигацию внутри сервиса без ручного управления серверным рендером.

### Минусы

- Между сервисами нет общего клиентского состояния; в эталоне состояния нет, межсервисное взаимодействие только на бэкенде.
- Нужна генерация типов из GraphQL-схемы.
- Нужны скелеты под ленивую загрузку.
- Нужен dev-сервер для проверки «цельного» приложения.

## Альтернативные предложения

- **Монолит** — быстро завести, но нет разнесения ресурсов, один большой бандл, падение одного узла бьёт по всему приложению; слабее отказоустойчивость и TTM на масштабе. Не выбираем.

- **Разные хосты и отдельные сервисы без выбранной модели** — плюсы по масштабированию и независимости есть, но при интеграции нужна оркестрация (лишний слой, задержки). Одна точка входа в бэкенд → у каждого сервиса дублируются CORS, кэш-политики, CSP, схемы и типы. Много дублирования и настройки межсервисного взаимодействия при малом выигрыше. Не выбираем.

- **Только клиентский рендер** — слабее по нагрузке и масштабированию относительно целей. Не подходит.

- **Ручное управление рендером** — снимает часть ограничений межсервисной связки, но по трудозатратам не дотягивает до уровня современных фреймворков. Не окупается.

- **Вынести бизнес-логику в Node (BFF)** — проще SSR и разбивка схем по цепочкам SSR–BFF–backend, но заметный проигрыш по производительности. Не выбираем.
