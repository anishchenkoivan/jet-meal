# ADR 03 единый формат данных: GraphQL

## Проблематика

Система продуктово единая, но бэкенд разделён на сервисы (изоляция данных, нагрузки и зон ответственности по бизнес-юнитам), клиентская часть — на отдельные приложения (микрофронтенды). Внешняя граница для клиентов — единый API gateway.

Требуется определить **единый формат обмена данными** между клиентом и сервером при условии:

- межсервисной интеграции на стороне бэкенда при обработке сценариев;
- различных потребностей клиентских приложений в объёме и структуре полезной нагрузки;
- удобного версионирования схем.

## Обоснование

Клиентскому приложению требуется **агрегированное представление данных** для интерфейса, при том что источники данных распределены по сервисам. Подход «много REST-ресурсов без агрегирующего слоя» приводит к росту числа запросов с клиента, дублированию логики согласования и повышению связанности клиента с деталями топологии сервисов.

## Решение

В качестве формата данных принимается **GraphQL**: согласованная схема типов и связей между ними, запрос на языке GraphQL. Внутри контура сервисов допустимы другие представления данных и протоколы; для клиентских приложений внешний контракт — эта схема и правила её изменения.

## Последствия

### Плюсы от внедрения

- Одна схема описывает контракт. Не нужно поддерживать отдельный набор JSON-форм под каждый сценарий.
- Клиент в запросе перечисляет нужные поля и вложенность. Можно обойтись без одного общего «толстого» DTO на всех.
- В схеме заданы типы, обязательность полей и связи между объектами. По ней удобно генерировать клиентские типы и проверки.
- Модель развивается через схему: новые поля, пометка старых как устаревших. Реже приходится плодить версии одного и того же ресурса.

### Минусы

- Ответ зависит от текста запроса. Один тип в схеме может вернуться с разным набором полей. Это нужно закладывать в код и тесты.
- Без правил изменений схема разрастается. Много опциональных полей и ветвей в графе усложняют поддержку.

## Альтернативные предложения

- **REST, JSON и OpenAPI.** Контракт — это набор эндпоинтов и фиксированных тел ответов. Нужна гибкая выборка под экран — либо тяжёлые объекты, либо новые пути и новые фрагменты в спецификации. Как единый формат описания данных для клиента не выбираем.

- **Protobuf (в т.ч. поверх gRPC-Web).** Сообщения заданы жёстко, сериализация компактная. Новый набор полей в ответе чаще всего означает новое сообщение или новую процедуру. Для одного общего гибкого контракта с клиентами это менее удобно, чем GraphQL-схема.

- **tRPC и аналоги.** Контракт следует из кода сервера и типов TypeScript. Для нескольких независимых клиентских кодовых баз хуже подходит как общий внешний описатель модели без привязки к языку. Слабее, чем явная схема GraphQL.

## Статус
